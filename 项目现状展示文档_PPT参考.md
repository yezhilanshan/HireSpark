# 职跃星辰（职跃星辰）项目现状展示文档（PPT参考）

## 一、项目定位与一句话介绍

职跃星辰 是一个面向求职训练场景的“AI 模拟面试与能力提升平台”。它通过“岗位化题目生成 + 多模态面试过程采集 + 结构化评估 + 复盘建议 + AI 问答辅助”的完整链路，把传统“凭感觉练面试”的低效方式升级为“有数据证据、有阶段目标、有针对性改进”的训练体系。

项目当前核心价值不是替代真人面试官，而是让用户在高频、低成本、可重复的训练中，快速发现短板并持续提升面试表现。

---

## 二、项目背景与问题定义

在真实求职场景中，大量候选人存在如下痛点：

1. 缺少高质量练习场。多数练习依赖朋友或同学，组织成本高，难以高频进行。
2. 反馈不具体。常见反馈停留在“表达不清楚”“回答太泛”，缺少可执行改进路径。
3. 训练过程不可追踪。每次练完就结束，无法形成历史趋势和能力画像。
4. 岗位导向不足。前端、后端、算法、产品等岗位的能力结构不同，但训练常常混用题目。
5. 面试焦虑无法量化。镜头状态、语速稳定性、注视偏移、异常波动等因素实际影响面试结果，但很少被系统记录和反馈。

职跃星辰 的设计目标是围绕这些问题构建结构化训练系统：

- 让训练过程标准化、可复用。
- 让反馈结果可量化、可解释。
- 让提升路径可追踪、可迭代。
- 让建议既基于知识库又结合用户历史数据。
- 让单场表现与阶段综合能力同时可见。

## 三、目标用户与核心使用场景

### 1. 目标用户

- 校招/社招求职者（技术岗、产品岗、算法岗等）。
- 需要高频练习、但缺少稳定面试陪练对象的人群。
- 希望系统化提升表达、项目陈述、系统设计能力的人群。
- 希望通过数据化方式管理求职节奏与复盘质量的人群。

### 2. 核心使用场景

- 场景 A：面试前 1-2 周冲刺训练，聚焦岗位核心题型与表达能力。
- 场景 B：多场模拟后，通过总结页识别重复短板，制定下一轮训练重点。
- 场景 C：围绕某一道题或某个能力薄弱点，在 AI 助手中进行多轮追问。
- 场景 D：复盘单场报告，定位“低分成因—证据来源—下一步动作”。

## 四、项目现状总览（当前可展示成果）

项目当前已具备“训练—评估—复盘—再训练”的完整主链路，属于可运行、可演示、可迭代阶段。关键现状如下：

1. 登录与会话机制可用，基础访问流程稳定。
2. 面试主流程可执行：进入训练、接收提问、完成回答、生成报告。
3. 历史记录可查看，可回到既往会话进行复盘。
4. 报告页面功能较完整，覆盖内容轴、语音、镜头、风险提示等维度。
5. 总结页支持近期趋势与综合画像展示。
6. AI 助手由轻量浮窗升级为独立功能区方向，支持多轮问答与历史沉淀。
7. RAG 已接入，问答可优先依据知识库并展示引用证据。
8. 后端具备健康检查能力，服务状态可观测。
9. 本地与服务器部署均已实践，具备基本工程可交付能力。

从成熟度看：

- 交互层：中高完成度。
- 数据闭环：中完成度。
- 评估策略：中完成度（可用，仍持续调优）。
- 工程稳定性：中完成度（可部署运行，仍有边界场景待优化）。

## 五、核心功能模块（适合拆成 PPT 多页）

## 1. 面试训练模块

该模块是平台入口，负责让用户快速进入岗位化模拟状态。

当前能力：

- 支持训练轮次/面试类型选择。
- 支持按题训练与策略生成提问。
- 支持摄像头和音频链路接入（受权限与安全上下文影响）。
- 过程事件可记录，供后续评估使用。

业务价值：降低开始练习门槛，并确保每次训练留下结构化数据。

## 2. 题库与知识组织模块

题库不是静态问答清单，而是训练可控性的核心资产。

当前能力：

- 按岗位与能力维度组织题目。
- 支持用户自主选择训练题，提升训练目的性。
- 题目内容可与知识库联动，减少泛化提问。

业务价值：让用户明确“今天练什么、为什么练这个”。

## 3. 面试报告模块（单场复盘）

报告模块是当前最具展示价值的模块之一。

输出内容：

- 综合分与轮次信息。
- 内容维度评分（表达清晰度、逻辑严谨性、架构推理等）。
- 语音表现与可执行建议。
- 镜头前表现与可执行建议。
- 风险提示与可信度拆解。

业务价值：把“答得不好”转化为“哪里不好、证据是什么、先改什么”。

## 4. 近期总结/综合画像模块（阶段复盘）

该模块从单场提升到阶段，帮助用户回答：

- 最近状态是上升、波动还是停滞？
- 当前岗位匹配度处于什么水平？
- 哪些问题在多轮中反复出现？
- 下一阶段训练重点是什么？

业务价值：从“看分数”升级为“看趋势 + 看短板 + 看行动方案”。

## 5. AI 问答助手模块

当前方向：从“右下角工具”升级为“正式功能区”。

目标能力：

- 多轮连续对话。
- 会话历史沉淀（可续聊、可回看）。
- RAG 优先回答并显示引用证据。
- 证据不足时明确标注“模型补充”，避免伪装成知识库事实。

业务价值：把问答助手变成训练中的“第二教练”。

## 6. 历史记录与回放模块

当前能力：

- 查看历史面试会话。
- 进入既往报告回看细节。
- 连接阶段总结观察成长轨迹。

业务价值：将离散练习转化为可持续资产。

## 六、技术架构现状（汇报重点：可扩展性）

## 1. 前端层

- 技术栈：Next.js + React + TypeScript + Tailwind + Recharts。
- 主要职责：训练流程交互、报告可视化、历史会话管理、助手功能区。
- 现状：核心页面已成型，正在持续优化主题一致性、异常态提示和交互效率。

## 2. 后端层

- 技术栈：Python 服务（接口层 + 业务服务层）。
- 主要职责：面试流程编排、评估计算、RAG 检索、助手生成、数据库读写。
- 现状：核心链路可用，可部署运行，性能与鲁棒性持续增强中。

## 3. 数据层

- 当前主要使用 SQLite（便于快速迭代与迁移）。
- 数据对象包括：面试会话、问答明细、评估结果、语音/镜头指标、助手会话历史等。
- 现状：支持产品演示与中小规模使用，后续可迁移 MySQL/PostgreSQL。

## 4. AI 能力层

- 大模型：用于评价总结、策略建议、助手回答生成。
- RAG：用于知识检索增强，降低幻觉风险。
- ASR/TTS：支持语音输入输出与主备策略。
- 多模态监测：采集镜头行为与过程事件，提升评估完整性。

## 七、项目优势（可作为“核心竞争力”页）

## 优势 1：闭环完整，不是单点工具

很多同类项目只做“问答”或“报告”之一。职跃星辰 已打通训练前、训练中、训练后链路，并能回流到下一轮训练。

## 优势 2：评估可解释，不是黑箱分数

系统不仅给分，还给依据、维度和建议，便于用户知道如何改善。

## 优势 3：岗位化训练能力明确

不同岗位采用不同题目与评价重点，避免“一个模板打天下”。

## 优势 4：RAG 优先，回答可追溯

助手回答可基于本地知识库并展示引用证据，增强可信度与可验证性。

## 优势 5：多模态视角更贴近真实面试

不仅看文本，还关注语音与镜头表现，评估更接近真实面试官感知。

## 优势 6：工程可落地，已完成部署实践

项目已经历本地/服务器部署、日志排障、联调修复，具备真实工程交付经验。

## 优势 7：迭代响应快

项目可基于反馈快速调整交互、评分口径、总结逻辑、助手流程，体现产品化能力。

## 八、当前挑战与客观限制（建议主动在答辩中说明）

1. 评分稳定性挑战：不同轮次、答题长度、语料质量会影响维度分一致性。
2. 数据采样质量挑战：设备、网络、权限、光线都会影响语音和镜头数据质量。
3. 响应速度挑战：助手链路包含检索与生成，极端情况下会出现延迟。
4. 环境一致性挑战：开发环境可用不等于公网环境稳定，需持续做部署标准化。
5. 数据一致性挑战：数据库迁移、服务重启、环境变量差异可能导致前后端表现不一致。

说明这些挑战不是减分项，反而体现项目工程认知真实、风险意识清晰。

## 十、标准业务闭环（流程图参考）

1. 用户登录后进入训练入口，选择岗位/题目/轮次。
2. 系统发起提问并记录语音与镜头过程数据。
3. 面试结束后生成结构化报告（分数、短板、建议）。
4. 用户在总结页查看趋势与综合画像，决定下阶段训练重点。
5. 用户进入 AI 助手围绕短板多轮追问，获取带引用建议。
6. 用户进行下一轮训练，验证改进是否生效。

这构成了“数据驱动成长飞轮”。

## 十一、项目方法论（可做“创新点提炼”页）

1. 结构化训练：训练对象（岗位、题目、轮次）先结构化，再评估。
2. 证据化反馈：先给证据，再给建议，提升用户信任与执行意愿。
3. 阶段化成长：不只看单场高低，更看跨场趋势与综合能力变化。

## 十二、后续优化路线（路线图）

## 短期（1-2 周）

1. 深色模式覆盖补齐与视觉一致性优化。
2. 评分口径说明增强，减少“分数为什么这样”的认知成本。

3. 数据层升级（向 MySQL/PostgreSQL 迁移方案）。
4. 自动化回归测试体系补齐（关键路径可重复验证）。

## 长期（2-6 月）

1. 岗位画像模板化与个性化学习路径推荐。
2. 时序分析增强（专注度曲线、表达节奏变化）。
3. 多环境部署标准化与监控告警完善。
4. 形成高校就业辅导版/企业训练版的产品分层能力。

## 十三、答辩高频问题与推荐回答

## Q1：和普通聊天机器人有什么区别？

A：我们是训练平台，不是通用聊天工具。核心是岗位化训练、可解释评估、阶段成长闭环，以及基于 RAG 的证据化问答。

## Q2：评分是不是主观？

A：系统采用“结构化指标 + 规则策略 + 模型解释”组合，不追求绝对客观，但追求可解释、可复核、可改进。

## Q3：为什么必须做 RAG？

A：仅靠模型内置知识存在时效与幻觉问题。RAG 能让回答锚定在项目知识库并提供引用，可信度更高。

## Q4：项目最难点是什么？

A：不是页面开发，而是评估口径稳定性、多模态数据质量和跨环境部署一致性。

## Q5：这个项目有产品化空间吗？

A：有，场景可扩展到高校就业指导、职业培训、企业内训等多个方向，核心能力具备迁移价值。
