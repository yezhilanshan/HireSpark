## 改进方向建议（详细版）

### 一、核心架构：三层分离设计

原文明确指出，当前系统不应改掉实时识别逻辑，而应在其上叠加两层：

| 层级 | 职责 | 技术实现 | 原文依据 |
|:---|:---|:---|:---|
| **实时交互层** | 低延迟字幕、打字机效果、AI即时插话/提示、判断用户说话状态 | 保留现有实时ASR（如FunASR-Realtime） | "你现在的实时识别逻辑，其实已经解决了一个最难的问题：低延迟交互。所以不要改掉它" |
| **聚合展示层** | 题目级文本拼接、草稿版完整回答展示给用户、重叠文本消解 | AnswerSession管理 + 文本去重拼接 + 轻量标点修复 | "把同一道题下的多个短分段，合成一条'完整回答'展示给用户" |
| **终稿确认层** | 高精度识别、修正术语/补全长句、为LLM评分/总结/追问提供可靠输入 | 后台高精ASR复写（如Qwen3-ASR-1.7B） | "在用户这道题答完后，把整段音频再送给后台ASR，生成最终文本" |

**关键原则**：前台依然快，用户看到的是完整回答（而非碎片），后端评分和总结用的是更干净、更准确的文本。

### 二、关键机制改进

#### 1. 双阈值停顿判断（核心改进）

原文强调"把'分段结束'与'整题结束'分开判断"是最关键的一点：

| 阈值类型 | 时长 | 作用 | 典型场景 |
|:---|:---|:---|:---|
| **短停顿阈值** | 900ms | 决定实时分段边界 | 用户正常说话中的呼吸停顿 |
| **长停顿阈值** | 4-6秒 | 判定整题回答结束，触发finalize流程 | 用户思考、换角度继续说、或真正答完 |

**设计原因**：原文指出"人说话经常会：想一下、换个角度继续说、分两三段表达同一个答案"，所以停1.2秒应切新分段但仍属同一题，停4秒才大概率答完。

**状态流转**：
- 停1.2秒 → 切新segment → 继续recording → 并入同一AnswerSession
- 停4秒 → 进入finalizing → 导出完整音频 → 触发高精ASR

---

#### 2. 题目级会话管理（AnswerSession）

原文建议引入"题目级回答会话"对象，而非只看单个分段：

**数据结构**：
```python
class AnswerSession:
    question_id: str           # 题目标识
    start_time: float          # 开始时间
    end_time: float | None     # 结束时间
    
    segments: list[dict]       # 每个实时分段（含文本、时间戳）
    merged_text_draft: str     # 拼接后的草稿文本
    final_text: str | None     # 高精ASR终稿（完成后回填）
    
    audio_chunks: list[bytes]  # 当前题目的原始音频块（连续PCM）
    status: str                # recording / paused_short / finalizing / finalized
```

**生命周期**：
1. 用户开始回答新题 → 创建AnswerSession（status=recording）
2. 每个实时分段识别结果挂载到该session，而非识别完就丢弃
3. 长停顿触发finalize → 导出`question_X_full_audio.wav` → 送高精ASR
4. 终稿返回 → 替换merged_text_draft → status=finalized

---

#### 3. 文本聚合器（去重拼接）

原文指出"只拼文本有两个问题"，因此需要专门处理：

**问题1：分段边界≠语义边界**
- 实时ASR按停顿分段，可能断在半句中间
- 粗暴拼接会产生别扭文本

**问题2：相邻分段重叠**
- 段1："我之前做过一个"
- 段2："一个推荐系统项目"
- 错误结果："我之前做过一个一个推荐系统项目"
- 正确结果（去重后）："我之前做过一个推荐系统项目"

**拼接规则**（原文建议）：
- 上一段末尾无句号 + 新段续句 → 直接接上，不强插标点
- 新段像完整新句 → 补句号后再拼接
- 检测重叠前缀 → 自动消解重复

**标点修复**：
- 草稿阶段：轻量修复（实时文本往往标点不准）
- 终稿阶段：完全替换，不依赖拼接规则

---

#### 4. 音频缓存策略

原文强烈建议"一定要保存整题原始音频"，这是高精ASR的基础：

**推荐方案A：保存整题连续PCM**
- 用户回答期间，所有原始音频块进入同一个buffer
- 题目结束时导出完整WAV
- 优势：简单，后续处理方便

**备选方案B：保存每个语音段再拼接**
- 每个分段单独存音频，结束时拼接
- 劣势：复杂度高，拼接处可能有瑕疵

**产出物**：
- `question_12_draft_text`（草稿文本）
- `question_12_full_audio.wav`（完整音频，用于高精复写）

### 三、状态机设计（详细）

原文建议设计4状态机管理前后端逻辑：

```
                    ┌─────────────────┐
                    │      idle       │ ← 初始状态，等待用户开始回答
                    └────────┬────────┘
                             │ 用户开始说话
                             ▼
                    ┌─────────────────┐
         ┌─────────│    recording    │ ← 正在回答，实时识别中
         │         │  (显示"正在识别...") │
         │         └────────┬────────┘
         │                  │ 短停顿>900ms
         │                  ▼
         │         ┌─────────────────┐
         └────────►│  paused_short   │ ← 短暂停顿，仍属当前题
                   │ (保持当前展示，等待恢复) │
                   └────────┬────────┘
                            │ 长停顿>4s 或 用户主动结束
                            ▼
                   ┌─────────────────┐
                   │   finalizing    │ ← 判定结束，等待高精复写
                   │ (显示"当前回答（整理中）") │
                   └────────┬────────┘
                            │ 高精ASR返回final_text
                            ▼
                   ┌─────────────────┐
                   │    finalized    │ ← 终稿已生成
                   │  (显示"最终回答")   │
                   └─────────────────┘
```

**状态转换条件**：

| 当前状态 | 转换条件 | 下一状态 | 操作 |
|:---|:---|:---|:---|
| recording | 短停顿>900ms | paused_short | 切新segment，更新draft |
| paused_short | 恢复说话 | recording | 继续追加 |
| paused_short | 长停顿>4s / 点击"回答完毕" / 说"我回答完了" | finalizing | 停止追加，导出音频，触发高精ASR |
| finalizing | 高精ASR返回 | finalized | 替换draft为final_text，送LLM分析 |
| finalized | 开始下一题 | idle | 创建新AnswerSession |

### 四、终稿替换与展示策略

原文建议给用户显示"两层状态"以避免突兀：

**回答过程中**：
- 界面标题："当前回答（实时识别中）"
- 内容：merged_text_draft（动态更新，去重拼接后）

**长停顿判定结束后**：
- 界面标题切换为："当前回答（整理中）"
- 内容：保持最后一版draft（或显示骨架屏）

**高精ASR完成后**：
- 界面标题切换为："最终回答"
- 内容：final_text整体替换draft
- 同步：将final_text发给LLM做评分/追问/总结

**用户体验优势**：用户不会觉得文本"突然变了"，而是感知到"从草稿到终稿"的合理过渡。

### 五、LLM输入分工建议

原文明确建议"不要让LLM直接吃实时碎片"：

| 链路 | 输入内容 | 负责功能 | 原因 |
|:---|:---|:---|:---|
| **实时链路** | 实时碎片文本 | 前端字幕、说话状态判断、简单实时提示 | 低延迟需求 |
| **题后链路** | final_text（终稿） | 面试评分、关键词提取、回答结构分析、STAR分析、生成追问、总结报告 | 避免LLM误判表达不完整、追问方向跑偏、总结重复或漏信息 |

**关键原则**：实时文本用于"交互体验"，终稿文本用于"智能分析"。

### 七、完整流程图（原文推荐）

```
开始一道题
  ↓
创建 AnswerSession（status=recording）
  ↓
持续接收音频块 → 同时进入audio_chunks缓存
  ↓
短静音 > 900ms：切一个segment，做实时ASR识别
  ↓
segment文本经过去重拼接 → 并入merged_text_draft
  ↓
前端展示"当前完整回答（草稿）"
  ↓
长静音 > 4s：判定该题结束（status→paused_short→finalizing）
  ↓
导出该题完整音频（question_X_full_audio.wav）
  ↓
送后台高精ASR（如Qwen3-ASR-1.7B）
  ↓
返回final_text
  ↓
替换merged_text_draft（status→finalized）
  ↓
把final_text发给LLM做评分/追问/总结
  ↓
进入下一题或生成面试报告
```

### 八、实施优先级（原文隐含逻辑）

| 阶段 | 任务 | 预期收益 |
|:---|:---|:---|
| **P0（立即）** | 保留现有实时逻辑，增加AnswerSession管理结构 | 不破坏现有体验，建立扩展基础 |
| **P1（短期）** | 实现双阈值判断 + 文本去重拼接器 + 轻量标点修复 | 用户看到完整回答而非碎片 |
| **P2（中期）** | 接入高精ASR复写流程 + 终稿替换机制 | 评分/总结质量显著提升 |
| **P3（长期）** | 基于终稿文本优化LLM评分算法、追问策略、报告生成 | 整体AI面试官体验质变 |

---

**总结**：这套改进方案的核心价值在于**分层解耦**——用实时层保体验，用聚合层保完整，用终稿层保质量，最终实现"又快又准"的AI面试系统。
